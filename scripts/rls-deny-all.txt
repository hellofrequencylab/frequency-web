# check:rls deny-all allowlist — tables that intentionally have RLS ENABLED but NO POLICY.
#
# RLS-on-no-policy = FAIL-CLOSED: the table is reachable ONLY by the service-role client
# (which bypasses RLS); anon/authenticated get nothing. It is the correct posture for a table
# no caller-role should ever touch directly — written/read via the admin client behind
# app-layer authz (server actions, crons, webhooks). This mirrors Supabase's own advisor,
# which rates `rls_enabled_no_policy` as INFO (safe), not a vulnerability.
#
# This file IS the documented posture. The guard (scripts/check-rls.mjs) fails a NEW table
# that has RLS off (real exposure) OR RLS-on-no-policy and is NOT listed here — forcing a
# conscious "is this deny-all on purpose?" decision at PR time. To intentionally add a
# service-role-only table, add its name below. One table per line; # comments allowed.
